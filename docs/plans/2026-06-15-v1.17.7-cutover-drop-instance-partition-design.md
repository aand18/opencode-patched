# Design: v1.17.7 cutover + drop `instance-state-partition.patch`

Date: 2026-06-15
Status: Approved (brainstorming complete; implementation plan to follow)
Branch/worktree: `chore/v1.17.7-cutover` at `~/projects/opencode-patched-v1.17.7-cutover`

## 1. Goal & context

Cut `opencode-patched` over from upstream **v1.17.4 → v1.17.7**, and retire
`instance-state-partition.patch`, because v1.17.7 ships the upstream fix for the
Question-tool hang we reported in issue
[#29772](https://github.com/anomalyco/opencode/issues/29772).

### What our patch did

`instance-state-partition.patch` (patch #5 in `patches/apply.sh`) fixed the
Question tool hanging on submit. Root cause: `InstanceStore.Service`
materialized **twice** per directory because the TCP listener
(`server.ts:startListener`) built its layer with `Layer.makeMemoMapUnsafe()`
(a fresh memoMap), while the in-process `Default` webHandler used the shared
process-global `memoMap`. Two memoMaps → two `InstanceStore.Service` →
two `Question.Service` pending-request maps. `Question.ask` stored in map A;
`Question.reply` arrived at map B and was rejected as `reply for unknown
request`, so the form never dismissed.

Our fix had two parts:
1. `server.ts:startListener` uses the shared `memoMap` instead of
   `Layer.makeMemoMapUnsafe()`, so both pipelines dedupe the `InstanceStore`.
2. `InstanceLayer.layer` made a stable reference (`= InstanceStore.layer`,
   bootstrap injected externally) so the shared memoMap keys correctly.

### What upstream did (and shipped in v1.17.7)

Commit `87c33b3` — `fix(plugin): reuse active server for client requests`
(merged to upstream `dev` **and** `production`; confirmed an ancestor of the
`v1.17.7` tag). It routes plugin SDK client calls through the **active
listener** (`Server.url`) over real HTTP, falling back to the in-process
`Server.Default().app.fetch` only when no listener exists. The plugin was the
empirically-observed trigger of the second `InstanceStore` in our diagnostic
(`serviceId=... fetch ... opencode-plugin ... creating instance`), so rerouting
it to the listener collapses everything onto one `InstanceStore` per directory.

**Key difference / risk signal:** upstream **kept** `makeMemoMapUnsafe()` at
`server.ts:125` in v1.17.7 (verified). They did *not* adopt the memoMap route.
There is even an unmerged upstream branch `share-listener-runtime`
(`c0d06deef`, by `rekram1-node`) that is byte-for-byte our approach — upstream
had it on the shelf and deliberately chose the narrow plugin bridge instead,
self-described as "intentionally a narrow bridge until v2 owns plugin clients."
So the architectural partition still exists; only the plugin path was rerouted
around it. This is **exactly why we verify before dropping** rather than assume.

### Why this needs care

`instance-state-partition.patch` was already dropped once before — at the 1.17
cutover, on the theory the rewritten instance/HTTP layer made it moot. That
theory was **wrong** (v1.17.2 `server.ts` still used `makeMemoMapUnsafe()` and
`instance-layer.ts` was byte-identical to the buggy 1.15 form), the Question
tool still hung, and the patch was re-ported. We will not repeat that mistake:
the drop is gated on a live reproduction, not on reading the diff.

## 2. Approach (chosen: B — branch-isolated cutover with a verify-before-drop gate)

All work happens on the isolated worktree `chore/v1.17.7-cutover` off
`release/v1.17`. `release/v1.17` does not move until the live repro passes.
The patch is **fully deleted** (not "parked"): git history + a documented drop
commit make rollback a one-commit `git revert`, consistent with repo convention
(prefill-fix, bus-eager-subscribe were removed, not parked).

Rejected alternatives:
- **A (in-place on release branch):** no isolation/review gate before the
  release branch changes.
- **C (keep patch parked as `.parked`):** redundant with git history; violates
  the delete-and-document convention. YAGNI.

## 4. Verification strategy (two gates)

### Gate 1 — upstream fix exists (automated; runs BEFORE we drop anything)

On a clean v1.17.7 checkout (no patches):
- Run upstream's regression test, confirmed present at
  `packages/opencode/test/server/httpapi-listen.test.ts:305` →
  `"plugin client requests reuse the listening server instance"`.
- Run `bun --cwd packages/opencode typecheck`.

This proves the bridge works in the released code before we remove our fix.

### Gate 2 — live reproduction (manual; the real bar)

Build a v1.17.7-patched binary with the 5 remaining patches (no
`instance-state-partition`), then reproduce the original scenario:
- `opencode serve` + a client (TUI and/or Telegram) **with at least one plugin
  loaded**. This is critical: the bug is plugin-triggered, so a repro without a
  plugin would not exercise `87c33b3` and would give a false pass.
- Trigger the Question tool repeatedly (prior bar: 11/11 clean submissions).

**Pass signal:** the form dismisses on every submit **and** zero
`reply for unknown request` / `reject for unknown request` warnings appear in
the server log across the run. (The old `serviceId` / `creating instance`
instance-count signal came from a local diagnostic build that v1.17.7 does not
have, so the signal here is behavioral + log-absence, not an instance count.
If a stronger signal is wanted, a temporary diagnostic log line can be added,
but behavioral + log-absence is the agreed bar.)

## 5. Rollback

The drop is one commit on the cutover worktree. If Gate 2 fails (e.g. a residual
partition from some non-plugin in-process `Default` consumer), `git revert`
restores the patch file + `apply.sh` entry. Its `server.ts` hunk still targets
the unchanged `makeMemoMapUnsafe()` line in v1.17.7, so it re-applies. Because
`release/v1.17` never moves until Gate 2 passes, the published release is never
exposed to a regression.

## 6. Process / sequencing (meta-flow)

1. Write this design doc → commit. *(this step)*
2. `writing-plans` → implementation plan (cutover runbook) in `docs/plans/`.
3. `opencode-launch` a **GPT-5.5** headless session to review design + plan +
   intended diff; fold in feedback.
4. `preparing-for-compaction` → persist durable context + resumption prompt.
5. Execute via **subagent-driven-development** (implementer / spec-reviewer /
   code-reviewer per task).
6. Merge to `release/v1.17`; confirm `sync-upstream` / `build-release` publishes
   v1.17.7-patched. The plan will also check whether CI *already* auto-built a
   v1.17.7-patched against the old (v1.17.4) patch set, and how that build
   fared.
