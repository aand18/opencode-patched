# v1.17.7 Cutover + Drop instance-state-partition.patch — Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans (or
> superpowers:subagent-driven-development) to implement this plan task-by-task.
> Repo-specific guidance: `.opencode/skills/patch-refresh.md`.

**Goal:** Retarget the `opencode-patched` stack from upstream v1.17.4 → v1.17.7,
drop `instance-state-partition.patch` (now fixed upstream by `87c33b3`, shipped
in v1.17.7), verify the Question-tool fix survives via upstream test + live
repro, then land on `main` and publish `v1.17.7-patched`.

**Architecture:** Patch-stack repo. `patches/apply.sh` applies an ordered list
of `.patch` files to a clean upstream clone; CI (`build-release.yml`) builds
4-platform binaries and publishes a GitHub release. All work happens on the
isolated worktree `chore/v1.17.7-cutover` at
`~/projects/opencode-patched-v1.17.7-cutover` (already created off
`release/v1.17`). `release/v1.17`/`main` do not move until the live repro passes.

**Tech Stack:** bash patch scripts, bun, effect, GitHub Actions.

**Key facts established during design (do not re-litigate):**
- Upstream fix `87c33b3` (`fix(plugin): reuse active server for client requests`)
  is an ancestor of tag `v1.17.7`. It routes plugin SDK calls through the active
  listener; the plugin was the trigger of the dual `InstanceStore` in our #29772
  diagnostic.
- v1.17.7 `server.ts:125` still uses `Layer.makeMemoMapUnsafe()` — upstream did
  NOT take our memoMap route; the architectural partition still exists, only the
  plugin path was rerouted. **This is why Gate 2 (live repro) is mandatory.**
- Upstream regression test exists at
  `packages/opencode/test/server/httpapi-listen.test.ts` →
  `"plugin client requests reuse the listening server instance"`.
- `main` is STALE (v1.16.2 stack incl. `prompt-loop-cache.patch`). `sync-upstream`
  auto-builds from `--ref main` every 8h and FAILS at `Applying
  prompt-loop-cache.patch`. Decision: land the cutover on `main` to make main the
  v1.17 line and fix auto-sync.
- `main` and `release/v1.17` diverged at `c122c58`; main has 4 obsolete commits,
  release/v1.17 has 6. "Land on main" = advance main's content to the cutover.

**Residual risk (accepted, per GPT-5.5 review claim #4):** `87c33b3` only covers
the observed plugin-triggered `opencode serve` scenario. Other in-process
`Server.Default().app.fetch` consumers still exist in v1.17.7
(`cli/cmd/run.ts:846/884`, `cli/tui/worker.ts:35`), but none run a second
`Default` pipeline concurrently with a listener for the same directory today
(`run` local modes have no listener; the TUI is strictly either listener-URL or
worker-`fetch`, never both — `cli/cmd/tui.ts:157-166`). The dropped patch would
have covered future/atypical mixed `Default`+listener consumers; if such a path
is introduced upstream later, the Question-tool partition could resurface and the
patch (or upstream's parked `share-listener-runtime` branch) would be the cure.
This is why Gate 2 is mandatory and why this risk is documented, not ignored.

**Temp/scratch:** use `/tmp/opencode/` for throwaway upstream checkouts.
**Working tree for edits:** `~/projects/opencode-patched-v1.17.7-cutover` (the
cutover worktree). All `git`/edit commands below assume this cwd unless stated.

### Task 4.2: Make `main` the v1.17 line

main and release/v1.17 diverged; main's 4 commits are obsolete v1.16.x work, and
main carries stale files release/v1.17 does not (e.g. `patches/prompt-loop-cache.patch`,
`patches/cache-aligned-compaction.patch`, `patches/mcp-reconnect.patch`).

> **DO NOT use `git merge -X theirs`** (review BLOCKING #1): `-X theirs` only
> resolves *conflicting* hunks — it preserves main-only files and non-conflicting
> main-side edits, so the resulting tree would NOT equal release/v1.17.

**Preferred (no force-push) — exact-tree merge.** Record release/v1.17 as a merge
parent but force main's entire tree to equal release/v1.17, verified twice:
```bash
git -C ~/projects/opencode-patched checkout main
git -C ~/projects/opencode-patched merge --no-ff --no-commit -s ours release/v1.17
git -C ~/projects/opencode-patched restore --source=release/v1.17 --staged --worktree :/
git -C ~/projects/opencode-patched diff --cached --exit-code release/v1.17 --   # MUST exit 0 (staged tree == release/v1.17)
git -C ~/projects/opencode-patched commit -m "Reconcile main onto the v1.17.7 patch line"
git -C ~/projects/opencode-patched diff --exit-code HEAD release/v1.17           # MUST exit 0 (commit tree == release/v1.17)
git -C ~/projects/opencode-patched push origin main
```
If either `--exit-code` check is non-zero, STOP — the trees differ; investigate
before pushing.

**Alternative (cleaner history, needs EXPLICIT user approval — force-push):**
`git branch -f main release/v1.17 && git push --force-with-lease origin main`.
Per global AGENTS.md, do not force-push without the user's explicit go-ahead.

**Verification:** both `git diff --exit-code` checks above pass (tree-identical).
